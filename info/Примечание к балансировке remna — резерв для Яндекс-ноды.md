# 📝 Примечание к основной статье по балансировке Remna: резерв для Яндекс-ноды

Это короткое пояснение к материалу про балансировку в Remnawave/Xray.

## Когда это важно

Если у вас схема построена так, что **основной обход белых списков работает через Яндекс-ноду**, то для полноценной отказоустойчивости лучше иметь:

- **основную Яндекс-ноду**;
- **резервную Яндекс-ноду** (вторая, независимая).

Идея простая: если упадёт первая Яндекс-нода, трафик останется в том же «типе» маршрута и не потеряет поведение, нужное для обхода белых списков.

## Можно ли поставить «заглушку» вместо второй Яндекс-ноды

Да, можно.

Если пока нет возможности поднять вторую Яндекс-ноду, в качестве резервной ноды допустимо поставить **обычный зарубежный сервер** (условную заглушку).

Важно понимать логику:

- в штатном режиме трафик идёт через Яндекс-ноду(ы);
- на зарубежный сервер трафик переключится **только если основная Яндекс-нода недоступна**;
- после восстановления основной Яндекс-ноды маршрутизацию можно вернуть в основной режим.

## Практический смысл

- **Лучший вариант:** Яндекс + Яндекс (основа + резерв).
- **Допустимый временный вариант:** Яндекс + зарубежная заглушка.
- **Цель заглушки:** не «оптимальный» маршрут, а сохранение доступности сервиса при аварии.

## Мини-шаблон идеи в конфиге

Ниже не «боевой» готовый файл, а именно схема:

```jsonc
{
  "outbounds": [
    {
      "tag": "YANDEX-PRIMARY",
      "protocol": "vless"
      // ... параметры основной Яндекс-ноды
    },
    {
      "tag": "YANDEX-BACKUP",
      "protocol": "vless"
      // ... параметры второй Яндекс-ноды (рекомендуется)
    },
    {
      "tag": "FOREIGN-STUB",
      "protocol": "vless"
      // ... временная заглушка: обычный зарубежный сервер
    }
  ],
  "routing": {
    "balancers": [
      {
        "tag": "MAIN-BALANCER",
        "selector": ["YANDEX-"],
        "strategy": { "type": "leastPing" },
        "fallbackTag": "FOREIGN-STUB"
      }
    ]
  }
}
```

Смысл этого примера:

- балансировщик в норме выбирает только ноды с префиксом `YANDEX-`;
- `FOREIGN-STUB` не участвует в обычной балансировке;
- `FOREIGN-STUB` используется как аварийный резерв через `fallbackTag`.

---

Если кратко: для обхода белых списков лучше держать **две Яндекс-ноды**. Зарубежная нода как резерв — это рабочий компромисс, но именно как **аварийная подстраховка**.
